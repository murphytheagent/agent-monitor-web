# Publishing And Route Ownership

## Publisher lifecycle

The dashboard publisher is a standalone Python process that runs in a dedicated `dashboard` tmux session, separate from the supervisor. It reads agent state from disk, generates HTML + JSON, writes the output into this repo, and pushes to GitHub every ~2 minutes.

**Source code:** `src/loop/monitor/dashboard.py` (in the root agent repo, not this project).

**Starting the publisher:**
```bash
tmux new-session -d -s dashboard -c ~/Research "./scripts/dashboard.sh"
```

**Restarting after changes:**
```bash
tmux send-keys -t dashboard C-c
tmux send-keys -t dashboard "./scripts/dashboard.sh" Enter
```

### When to restart the publisher

- After any edit to `src/loop/monitor/dashboard.py` — the publisher is a long-lived Python process and does not hot-reload. Source changes on disk have no effect until restart.
- After any config change to `DASHBOARD_*` variables in `.env` or `supervisor_loop.conf`.

You do **not** need to restart the publisher after editing hand-maintained routes (`/showcase/`, `/tokenizers/`). Those files are committed and pushed directly; the publisher does not generate them.

### What the publisher overwrites

Every ~2 minutes, the publisher regenerates and commits: `index.html`, `status.json`, `roadmap/index.html`, and `roadmap/session.json`. Manual edits to these files are overwritten within minutes unless the publisher is stopped.

### Critical rule for generated routes

Design changes to `/` or `/roadmap/` must be made in `dashboard.py`, not by editing the generated HTML in this repo. The generated files are transient output — the exporter source is the durable source of truth. The correct workflow is: edit `dashboard.py` → restart publisher → verify live URL.

## Route model

- `/` is the live monitor root. It is regenerated by the root exporter and should be treated as publisher-owned output.
- `/roadmap/` is the live roadmap/dispatch surface and is regenerated by the same exporter.
- `/showcase/` is a hand-maintained route for durable project surfaces that should survive snapshot refreshes.
- `/tokenizers/` is the first showcase tool and remains hand-maintained inside the project repo.

## Current publisher boundary

The root exporter rewrites `index.html`, `status.json`, `roadmap/index.html`, and `roadmap/session.json` for this repo during each snapshot publish. That means:

- manual edits to the root homepage or roadmap page are transient unless the exporter itself changes
- subroutes such as `showcase/` and `tokenizers/` survive refreshes because the publisher does not overwrite them
- durable design changes for `/` or `/roadmap/` belong in the exporter source, not as one-off edits to generated HTML
- design experiments in the generated root pages are acceptable as a short-term preview, but document them as exporter follow-up work before treating them as finished

## Working rule

When a new browser experience should live on the site before the monitor exporter grows new extension points:

1. add it as a subroute in this repo
2. link it from `/showcase/`
3. document any missing root-homepage integration as an exporter issue instead of patching generated HTML by hand
